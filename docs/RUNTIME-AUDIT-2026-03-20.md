## Findings

### P1. Signature header payload is transformed before transport

Files:
- [src/infrastructure/agent_identity.rs](/Users/fullfrontend/Jobs/A%20-%20Full%20Front-End/retaia-workspace/retaia-agent/src/infrastructure/agent_identity.rs:97)

Observed behavior before fix:
- the detached OpenPGP ASCII-armored signature was generated correctly
- before returning it, the implementation rewrote line breaks:
  - `\r` removed
  - `\n` replaced by literal `\\n`

Remediation:
- the detached ASCII-armored signature remains generated by Sequoia
- the runtime no longer emits a backslash-escaped `\\n` representation
- the HTTP transport now uses a compact armor form without raw line breaks, which is header-safe and avoids the previous non-contractual escaping layer

Spec reference:
- [specs/workflows/AGENT-PROTOCOL.md](/Users/fullfrontend/Jobs/A%20-%20Full%20Front-End/retaia-workspace/retaia-agent/specs/workflows/AGENT-PROTOCOL.md:121)

Current status:
- fixed in agent runtime
- residual note: the specs still describe verbatim multiline ASCII armor in a header, which standard HTTP headers cannot carry directly; Core and Agent must share the same compact transport rule until the contract text is clarified

Recommended follow-up:
- codify the compact header transport in specs, or move the signature out of headers if raw multiline armor must remain the normative wire format
- add one end-to-end verifier test using the exact transmitted header value

### P2. Private OpenPGP key is persisted in plaintext app storage

Files:
- [src/infrastructure/agent_identity.rs](/Users/fullfrontend/Jobs/A%20-%20Full%20Front-End/retaia-workspace/retaia-agent/src/infrastructure/agent_identity.rs:154)

Observed behavior before fix:
- `openpgp_private_key` was stored directly in `identity.json`
- Unix permissions were reduced to `0600`
- no equivalent hardening existed for non-Unix targets

Remediation:
- `openpgp_private_key` is no longer stored in `identity.json`
- the private key is now persisted in the OS secret store
- legacy plaintext identity files are migrated automatically on next load, then rewritten without the private key

Spec reference:
- [specs/workflows/AGENT-PROTOCOL.md](/Users/fullfrontend/Jobs/A%20-%20Full%20Front-End/retaia-workspace/retaia-agent/specs/workflows/AGENT-PROTOCOL.md:82)

Current status:
- fixed in agent runtime
- key theft from config directory alone is no longer sufficient to impersonate the agent

Recommended follow-up:
- run one packaging check per target OS to validate secret-store availability in the final app runtime

## Closed Gaps

These previously known gaps are now addressed on this branch:

- runtime no longer uses placeholder `job_id` as `X-Retaia-Agent-Id`
- technical bearer can be minted from configured `client_id + secret_key`

- current review did not uncover a runtime bug in `claim/heartbeat/submit/upload` sequencing after the conformance refactor
- generated client/doc noise is large; future reviews should isolate generated diff from runtime diff earlier
- secret-store availability still depends on correct runtime packaging and session setup on each target platform
